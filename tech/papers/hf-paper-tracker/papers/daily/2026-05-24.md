### [T1] KVServe: Service-Aware KV Cache Compression for Communication-Efficient Disaggregated LLM Serving
- URL: https://arxiv.org/abs/2605.13734
- 重要度: ⚡High
- 要約:
  1. [課題] 分離型 LLM serving では KV cache 転送が通信ボトルネックになり、固定圧縮設定では workload や SLO 変動に対応しにくい。
  2. [手法] 圧縮方式を再構成可能な戦略空間として統合し、Bayesian profiling と online bandit controller で状況ごとに最適な圧縮プロファイルを選ぶ KVServe を提案した。
  3. [結果] vLLM 統合評価で PD-separated serving の JCT を最大 9.13 倍、KV-disaggregated の TTFT を最大 32.8 倍改善した。
- 一言メモ: PCIe 前提の複数 GPU 推論で通信が支配的になりやすく、EP 構成の KV 転送最適化としてかなり直接的。

### [T1] WorldKV: Efficient World Memory with World Retrieval and Compression
- URL: https://arxiv.org/abs/2605.22718
- 重要度: 🔵Medium
- 要約:
  1. [課題] 長いロールアウトでは full KV attention が整合性維持に有効でも、メモリと attention コストが線形増大してリアルタイム性を失う。
  2. [手法] evict 済み KV chunk を GPU/CPU に退避しつつ、scene-relevant な chunk だけを再挿入する World Retrieval と、key 類似度で冗長 token を削る World Compression を組み合わせた。
  3. [結果] 学習不要で full-KV に近い world consistency を保ちながら、約 2 倍の throughput と固定予算下で 2 倍の履歴保持を実現した。
- 一言メモ: 世界モデル向けだが、長文 RAG や継続対話でも「全部持つ代わりに必要履歴だけ戻す」設計の参考になる。

### [T3] Efficient Agentic Reasoning Through Self-Regulated Simulative Planning
- URL: https://arxiv.org/abs/2605.22138
- 重要度: ⚡High
- 要約:
  1. [課題] agentic LLM は必要以上に長い推論を回しがちで、token 消費に対して精度向上が安定しない。
  2. [手法] world model に基づく simulative reasoning、計画深度を決める self-regulation、反応的 execution を分離した SR^2AM を提案した。
  3. [結果] 30B モデルで 120B〜1T 級相当の Pass@1 に迫りつつ、既存 agentic LLM 比で reasoning token を 25.8〜95.3% 削減した。
- 一言メモ: 金融エージェントの plan 深度を案件別に制御できれば、精度を維持しつつレイテンシとコストをかなり削りやすい。

### [T2/T3] ClinSeekAgent: Automating Multimodal Evidence Seeking for Agentic Clinical Reasoning
- URL: https://arxiv.org/abs/2605.20176
- 重要度: ⚡High
- 要約:
  1. [課題] 既存の臨床エージェントは curated evidence 前提が多く、実運用で必要な生データからの能動的証拠収集を扱えていなかった。
  2. [手法] 医療 KB 検索、EHR 走査、画像ツール呼び出しを通じて仮説更新しながら証拠収集する inference-time agent と、その軌跡蒸留パイプラインを構築した。
  3. [結果] text-only EHR と multimodal clinical task の両方で大幅改善し、蒸留した 35B-A3B モデルも強いベースラインを大きく上回った。
- 一言メモ: 社内文書・XBRL・決算資料をまたぐ金融 RAG でも、受動検索より evidence-seeking 軌跡の学習が効く可能性が高い。

### [T2/T3] Spreadsheet-RL: Advancing Large Language Model Agents on Realistic Spreadsheet Tasks via Reinforcement Learning
- URL: https://arxiv.org/abs/2605.22642
- 重要度: ⚡High
- 要約:
  1. [課題] 汎用 LLM をプロンプトだけで spreadsheet 操作させる方式は、実務的な多段ワークフローで失敗しやすい。
  2. [手法] Excel 環境上の Spreadsheet Gym、start-goal シート自動収集、finance を含む Domain-Spreadsheet を使う RL fine-tuning 枠組みを提案した。
  3. [結果] SpreadsheetBench と domain-specific ベンチの両方で Pass@1 を大きく改善し、実務寄りタスクでの一般化を示した。
- 一言メモ: 財務モデリングやポートフォリオ分析の表計算自動化に直結し、金融エージェントの作業実装に最も近い。

### [T3/T4] ACC: Compiling Agent Trajectories for Long-Context Training
- URL: https://arxiv.org/abs/2605.21850
- 重要度: ⚡High
- 要約:
  1. [課題] 長文 reasoning 学習データの作成は高コストで、通常の agent SFT では tool response が mask され文脈統合能力の監督が不足する。
  2. [手法] 検索・ソフトウェア開発・DB クエリの agent 軌跡を、質問と散在する観測証拠を束ねた long-context QA にコンパイルする ACC を提案した。
  3. [結果] Qwen3-30B-A3B を 235B 級に近い MRCR/GraphWalks 水準まで押し上げつつ、一般能力を維持した。
- 一言メモ: 自社エージェントの実行ログをそのまま長文 SFT データに変換できるので、RAG とツール利用を同時に鍛えやすい。

### [T3] π-Bench: Evaluating Proactive Personal Assistant Agents in Long-Horizon Workflows
- URL: https://arxiv.org/abs/2605.14678
- 重要度: 🔵Medium
- 要約:
  1. [課題] 既存ベンチは、長期対話で徐々に顕在化する hidden intent を先回りして処理する proactive assistance を十分評価できていない。
  2. [手法] hidden user intent、inter-task dependency、cross-session continuity を含む 100 の multi-turn task から成る π-Bench を構築した。
  3. [結果] 現状モデルは task completion と proactivity の両立が難しく、特に prior interaction を踏まえた先回り支援が弱いことを示した。
- 一言メモ: 投資判断支援ではユーザーが制約や評価軸を後出ししがちなので、proactive 性能の定量評価軸として有用。

### [T3] TerminalWorld: Benchmarking Agents on Real-World Terminal Tasks
- URL: https://arxiv.org/abs/2605.22535
- 重要度: 🔵Medium
- 要約:
  1. [課題] 既存 terminal agent ベンチは人工的なタスクが多く、実際の開発者ワークフローの多様さを十分捉えられていない。
  2. [手法] 80,870 件の実世界 terminal recording を自動解析し、1,530 の検証済みタスクと 200 の高品質 Verified subset を作成した。
  3. [結果] 8 つの frontier model と 6 つの agent を評価したところ、最高 pass rate は 62.5% に留まり、既存ベンチとの相関も弱かった。
- 一言メモ: MCP や社内ツールを叩く開発支援エージェントでは、整ったベンチより現場ログ由来タスクの方が実態に近い。

### [T3] Maestro: Reinforcement Learning to Orchestrate Hierarchical Model-Skill Ensembles
- URL: https://arxiv.org/abs/2605.22177
- 重要度: 🔵Medium
- 要約:
  1. [課題] 単一 LLM と固定ロジックに依存するエージェントは、モデルやスキルの補完関係を活かし切れない。
  2. [手法] 外部 expert の呼び出し、model-skill pair の選択、終了判断を逐次意思決定として学習する 4B orchestrator を outcome-based RL で訓練した。
  3. [結果] 10 ベンチ平均で GPT-5 と Gemini-2.5-Pro を上回り、未学習の model や skill を追加しても一般化した。
- 一言メモ: 金融タスク別に reasoning モデル、表計算スキル、RAG 検索器を切り替える設計の有力な参照例。

### [T4] DelTA: Discriminative Token Credit Assignment for Reinforcement Learning from Verifiable Rewards
- URL: https://arxiv.org/abs/2605.21467
- 重要度: ⚡High
- 要約:
  1. [課題] RLVR では応答単位の報酬がどの token に効いているかが曖昧で、書式のような高頻度 token に更新が引っ張られやすい。
  2. [手法] token-gradient の正負側 centroid をより識別的にする係数を学習し、共有的な更新方向を抑える DelTA を提案した。
  3. [結果] 数学 7 ベンチで同規模の強ベースラインを上回り、コード生成や別 backbone にも一般化した。
- 一言メモ: 構造化出力スキーマ品質を上げたいなら、正答だけでなく schema 準拠に効く token へ報酬を寄せる発想が重要。

### [T4] Unsupervised Process Reward Models
- URL: https://arxiv.org/abs/2605.10158
- 重要度: ⚡High
- 要約:
  1. [課題] PRM は有効だが、推論ステップごとの専門家アノテーション収集コストが高く大規模化しにくい。
  2. [手法] 次 token 確率から最初の誤り位置を評価する教師なしスコア関数で uPRM を学習し、最終解答ラベルも不要にした。
  3. [結果] 誤り位置同定で LLM-as-a-Judge を最大 15pt 上回り、test-time scaling と RL reward の両方でも supervised PRM 級の効果を示した。
- 一言メモ: 金融 reasoning データの逐次ラベル付けは高価なので、教師なし PRM はアライメントの現実解になりうる。

### [T2/T4] Training Large Language Models to Predict Clinical Events
- URL: https://arxiv.org/abs/2605.12817
- 重要度: 🔵Medium
- 要約:

  2. [手法] 過去文脈と未来イベントに関する自然言語質問、後続記録で解決されるラベルからなる予測データを作り、LoRA adapter で学習した。

- 一言メモ: 金融でも時系列メモや開示文書から「将来イベント質問」を作る形で、少量 LoRA による予測特化が試せる。

### [T4] LoREnc: Low-Rank Encryption for Securing Foundation Models and LoRA Adapters
- URL: https://arxiv.org/abs/2605.13163
- 重要度: 🔵Medium
- 要約:
  1. [課題] foundation model と LoRA adapter の配布では、モデル復元や知財漏洩リスクがある一方で、既存防御は再学習や元データ前提になりがちだった。
  2. [手法] FM 重みの支配的 low-rank 成分を抑制し、正規利用者の adapter で欠落情報を補償する training-free 保護法 LoREnc を提案した。
  3. [結果] 認可外利用では出力が構造崩壊し、認可利用では元性能を回復しつつ、追加計算は 1% 未満に抑えた。
- 一言メモ: 金融向けドメイン LoRA を社内外へ展開する運用では、軽量な adapter 保護の選択肢として覚えておきたい。

### [T3] Rule2DRC: Benchmarking LLM Agents for DRC Script Synthesis with Execution-Guided Test Generation
- URL: https://arxiv.org/abs/2605.15669
- 重要度: 🔵Medium
- 要約:
  1. [課題] コード生成エージェントの評価は類似度依存になりやすく、実行正しさや候補選別の実運用性を捉えにくかった。
  2. [手法] 1,000 タスク・13,921 レイアウトの実行ベース評価基盤 Rule2DRC と、実行結果から識別的テストを作る tester agent を導入した。
  3. [結果] 正解レイアウトを agent 入力に渡さずに機能正しさを測定でき、Best-of-N の候補選別性能も大きく改善した。
- 一言メモ: 金融ツール実行エージェントでも、コードや手順の見た目ではなく execution outcome で評価する設計が重要だと分かる。
