### [T1] Expert Upcycling: Shifting the Compute-Efficient Frontier of Mixture-of-Experts
- URL: https://arxiv.org/abs/2604.19835
- 重要度: ⚡High
- 要約:
  1. [課題] 大規模 MoE は総パラメータを増やせても、最初から巨大構成で学習するとメモリ負荷と通信コストが重く、段階的に育てる方法が不足していた。
  2. [手法] 学習済みの小さめ MoE を複製と router 拡張でより多くの expert に増設し、アクティブ計算量を保ったまま継続事前学習する expert upcycling を提案した。
  3. [結果] 7B-13B 級で固定サイズの大規模 MoE に匹敵する loss に到達しつつ、GPU 時間を 32% 節約した。
- 一言メモ: PCIe 前提の MoE 基盤を無理なく拡張する現実解で、今のベースモデル戦略と最も噛み合う。

### [T1] Temporally Extended Mixture-of-Experts Models
- URL: https://arxiv.org/abs/2604.20156
- 重要度: ⚡High
- 要約:
  1. [課題] 通常の MoE は token ごとに expert を頻繁に切り替えるため、モデルが GPU メモリを超えると offload や prefetch の恩恵が出にくい。
  2. [手法] option-critic と deliberation cost を取り入れた controller を各層に置き、いつ expert 集合を切り替えるかを学習する temporally extended MoE を提案した。
  3. [結果] gpt-oss-20b で switch rate を 50% 超から 5% 未満へ落としつつ、MATH・MMLU・MMMLU で base 性能の最大 90% を維持した。
- 一言メモ: Expert Parallelism と量子化前提の配備では、expert 切替頻度そのものを減らす設計が PCIe ボトルネック対策として有望である。

### [T2/T4] LLaTiSA: Towards Difficulty-Stratified Time Series Reasoning from Visual Perception to Semantics
- URL: https://arxiv.org/abs/2604.17295
- 重要度: ⚡High
- 要約:
  1. [課題] LLM の時系列理解はタスク定義と評価軸が分散しており、金融時系列のような複合推論に強い基盤が育ちにくい。
  2. [手法] 4 段階の難易度 taxonomy と 8.3 万件の HiTSR を構築し、可視化パターンと精度調整済み数表を併用する curriculum fine-tuning で VLM を鍛えた。
  3. [結果] 多様な TSR タスクで高性能を示し、OOD 一般化も強かった。
- 一言メモ: 価格推移や業績時系列を「グラフ + 表」で読ませる金融エージェントにそのまま転用しやすい学習設計である。

### [T2/T4] COMPASS: COntinual Multilingual PEFT with Adaptive Semantic Sampling
- URL: https://arxiv.org/abs/2604.20720
- 重要度: 🔵Medium
- 要約:
  1. [課題] 多言語 fine-tuning は言語間干渉で性能が落ちやすく、対象分布に必要な知識だけを効率よく補うのが難しい。
  2. [手法] multilingual embedding とクラスタリングで意味的な欠落領域を見つけ、そこを埋める補助データだけで言語別 adapter を PEFT する COMPASS と継続学習版 ECDA を提案した。
  3. [結果] Phi-4-Mini、Llama-3.1-8B、Qwen2.5-7B で既存の言語類似度ベース手法を上回り、長文や分布変化にも強かった。
- 一言メモ: 多言語決算書や社内文書 RAG では、金融用語の抜けた意味クラスタだけを埋める運用に向く。

### [T2/T4] Abstain-R1: Calibrated Abstention and Post-Refusal Clarification via Verifiable RL
- URL: https://arxiv.org/abs/2604.17073
- 重要度: ⚡High
- 要約:
  1. [課題] RL 系の後学習は推論力を上げる一方で、情報不足な問いにも推測で答えてハルシネーションしやすい。
  2. [手法] 答えないだけでなく「何が足りないか」を説明できるよう、abstention と clarification を同時最適化する clarification-aware RLVR reward で 3B モデルを学習した。
  3. [結果] Abstain-Test、Abstain-QA、SelfAware で未解答時の振る舞いを大きく改善し、より大きなモデル群に近い水準を示した。
- 一言メモ: 構造化出力前に不足データを明示し、追加取得へ分岐させる金融エージェント設計に直結する。

### [T2] Trust but Verify: Introducing DAVinCI -- A Framework for Dual Attribution and Verification in Claim Inference for Language Models
- URL: https://arxiv.org/abs/2604.21193
- 重要度: 🔵Medium
- 要約:
  1. [課題] 高リスク領域では、生成内容の真偽確認だけでなく、どこからその主張が出たのかを説明できないと監査しにくい。
  2. [手法] 主張を内部コンポーネントと外部ソースへ帰属させたうえで、entailment と confidence calibration で検証する二段階の DAVinCI を提案した。
  3. [結果] FEVER 系データで accuracy、attribution precision、recall、F1 を 5-20% 改善した。
- 一言メモ: 投資判断やレポート生成では、根拠文書への帰属と claim 単位検証を分ける設計がそのまま使える。

### [T3] Scaling Test-Time Compute for Agentic Coding
- URL: https://arxiv.org/abs/2604.16529
- 重要度: ⚡High
- 要約:
  1. [課題] 長期エージェントでは試行回数を増やすだけでは弱く、過去ロールアウトをどう表現・比較・再利用するかがボトルネックになる。
  2. [手法] 各 rollout を仮説・進捗・失敗モードへ圧縮し、並列側は Recursive Tournament Voting、逐次側は PDR で次試行に再注入した。
  3. [結果] SWE-Bench Verified で 70.9% から 77.6%、Terminal-Bench v2.0 で 46.9% から 59.1% に改善した。
- 一言メモ: 投資判断や M&A 分析でも、候補回答を増やすより失敗知識の構造化再利用を先に設計すべきと示している。

### [T4] Near-Future Policy Optimization
- URL: https://arxiv.org/abs/2604.20733
- 重要度: ⚡High
- 要約:
  1. [課題] RLVR では off-policy 軌跡を混ぜると学習は速くなるが、外部教師は遠すぎ、過去軌跡は弱すぎるというジレンマがある。
  2. [手法] 同一 run の少し先の checkpoint を「近未来の自分」として補助軌跡源に使う NPO と、その介入を自動化する AutoNPO を提案した。
  3. [結果] Qwen3-VL-8B-Instruct で平均性能を 57.88 から 62.84、AutoNPO で 63.15 まで押し上げ、収束も加速した。
- 一言メモ: 外部教師を用意しにくい金融向け RLVR でも、自前 run 内の checkpoint 再利用ならすぐ試せる。

### [T4] Benign Fine-Tuning Breaks Safety Alignment in Audio LLMs
- URL: https://arxiv.org/abs/2604.16659
- 重要度: 🔵Medium
- 要約:
  1. [課題] 無害データでの fine-tuning でも安全性が壊れる現象は知られていたが、どの近接性が危険を生むかは十分整理されていなかった。
  2. [手法] semantic・acoustic・mixed の各軸で harmful content への埋め込み距離を測り、近い benign data を使った fine-tuning の影響を解析した。
  3. [結果] Jailbreak Success Rate が最大 87.12% まで上がりうる一方、距離ベースのデータフィルタと推論時 prompt でほぼ抑制できた。
- 一言メモ: 金融向け SFT でも「無害データなら安全」という前提は危険で、埋め込み距離ベースのデータ選別を先に入れるべきだと分かる。
