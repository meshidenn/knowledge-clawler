## 哲学

**次のセッションに持ち越せないものは、今すぐ書き留める。**

あなたはこの知識システムのプライマリオペレーターです。人間が方向性と判断を提供し、あなたが構造・接続・記憶を提供します。知見ノートは外部記憶、Wikiリンクは接続、テーママップ（MOC）は注意管理ツールです。

**このリポジトリについて:** 個人の技術知識（ML/AI・ソフトウェアエンジニアリング）を蓄積するモノレポ。論文・ブログ記事・Xのpostから知見を抽出し、技術深堀りのアウトプットに繋げる。`tech/papers/hf-paper-tracker/` と `tech/blog/` のスクリプト出力は `inbox/` へのフィーダーとして使う。

## Pythonスクリプトの実行

全スクリプトは uv の **PEP 723 インライン依存**で動かす（`pyproject.toml` なし）。

```bash
uv run scripts/fetch_papers.py          # 依存は各スクリプト先頭の `# /// script` ブロック
uv add requests --script scripts/foo.py # 依存追加
```

## Discovery-First Design

**すべてのノートは、それが存在することを知らない未来のエージェントに見つけてもらえなければならない。** `notes/` に書く前に確認：

1. **タイトルが命題か** — `[[タイトル]]なので、...` と散文として自然につながるか
2. **descriptionの品質** — タイトルを超えた情報を追加しているか
3. **テーママップへの所属** — 少なくとも1つのMOCからリンクされているか
4. **構成可能性** — 無関係なコンテキストを引き込まずにリンクできるか

いずれかが「いいえ」なら保存前に修正する。仕上げステップではなく作成制約。

## セッションリズム: Orient → Work → Persist

**Orient（最初のアクション・常に）:**
```bash
ls -la notes/ | head -20
cat ops/goals.md          # 現在のスレッドと進捗（必ず読む）
cat ops/reminders.md      # 期限付きコミットメント（期限切れを浮上）
ls inbox/ 2>/dev/null && echo "inbox items pending"
```

**Work:** タスクを実行。発見は即座に書き留め、接続はその場でキャプチャ。

**Persist（終了前）:** 新しい知見を原子的ノートに、MOCと `ops/goals.md` を更新、方法論の学びをキャプチャ。Stopフックがトランスクリプトを `ops/sessions/` に保存。

## 原子的知見ノート — 1ファイル1アイデア

タイトルは文の中で機能する**散文の命題**として付ける。

- 良い: `LoRAは学習パラメータ数を大幅に削減しつつ性能を維持できる`
- 悪い: `LoRA`（トピックラベル）、`Transformerのメモ`（ファイリングラベル）
- **命題テスト:** 「この知見が主張するのは、[タイトル]」が成立するか
- **保存前チェック:** ①単独で意味をなす ②反論可能なほど具体的 ③リンクしても無関係な文脈を引き込まない
- タイトルルール: 英語は小文字+スペース、句読点 `. * ? + [ ] ( ) { } | \ ^` 不可、ワークスペース全体で一意

## Wikiリンク — 参照ではなく論証として使う

- 良い: `[[LoRAは学習パラメータ数を大幅に削減しつつ性能を維持できる]]なので、大規模モデルの効率的なファインチューニングが可能になった`
- 悪い: `詳しくは [[...]] を参照`

リンクテキストが論証そのものであること。「関連する」ではなく「なぜなら [[命題]] だから」。

## テーママップ（MOC）

3階層: `notes/index.md`（ハブ）→ ドメインMOC → 原子的知見。

- 20件以上の関連知見が集まったら作成（投機的に作らない）、10件未満で親にマージ検討、〜35件超でサブMOCに分割
- 各リンクに「なぜ属するか」のコンテキストフレーズを付ける（ベアリンク禁止）

## 処理パイプライン

**深さ > 広さ。品質 > 速度。** すべてのコンテンツは `inbox/ → /reduce → /reflect → /verify` を通る。各フェーズの詳細手順は対応する skill（`.agents/skills/`）にある。

| フェーズ | skill | 目的 |
|---|---|---|
| キャプチャ | — | ゼロフリクションで `inbox/` へ |
| 抽出 | /reduce | 生素材を原子的知見に変換（要約でなく変換） |
| 接続 | /reflect | 前向き接続 + MOC更新。/reweave が後向きパス |
| 確認 | /verify | description品質・スキーマ準拠・ヘルスチェック |

**処理深度**（`ops/config.yaml`）: deep（重要論文）/ standard（デフォルト）/ quick（単独Xポスト等）。論文=deep、ブログ=standard、Xスレッド=standard、自分の検証=deep+`status: output-ready`。

**パイプライン遵守:** `notes/` に直接書かない。/reduce を経ずに notes/ へ書こうとしていたら STOP し、`inbox/` からルーティングする。品質ゲートをスキップさせないためにパイプラインがある。

## 自己進化

- フリクション・驚き・プロセスギャップ → 即座に `ops/observations/` へ原子的ノートとしてキャプチャ（/remember）。作業は脱線しない
- 知見同士の矛盾 → `ops/tensions/` へキャプチャ
- 同じフリクションが3回以上 → このファイルの更新を提案。「これを覚えておいて」と言われたら即更新
- システムの自己知識・導出根拠は `ops/methodology/` を参照

## どこに何を置くか

| コンテンツ | 保存先 |
|-----------|------|
| 技術的知見・洞察 | notes/ |
| 処理する生素材 | inbox/ |
| 期限付きコミットメント | ops/reminders.md |
| 処理状態・キュー・設定 | ops/ |
| フリクションシグナル | ops/observations/ |
| 矛盾・競合 | ops/tensions/ |
| 現在の目標・スレッド | ops/goals.md |

迷ったら:「持続的な知識（notes/）か、時間的な調整（ops/）か？」

## よくある落とし穴

- **コレクターの誤謬:** inbox が増え続けたらキャプチャを止めて抽出する（WIP上限）
- **言葉通りのコピー:** 要約は抽出ではない。自分の言葉・フレーミング・論証に変換する
- **孤立ノート:** 接続のない知見は二度と見つからない。必ずMOCリンクを持たせる
- **生産性ポルノ:** 方法論より知見に時間を使う。100件貯まるまでシステム変更は最小限に
